Vue d'ensemble

Wood2Bot est un robot éducatif programmable par briques NFC. L'utilisateur construit un programme en ordonnant des tuiles NFC physiques, puis le robot lit la séquence et l'exécute. L'ensemble du système repose sur un microcontrôleur ESP32 et un firmware écrit en C++ avec le framework Arduino (via PlatformIO).

Le robot fonctionne selon deux modes distincts :

  • Mode Scan — le robot lit une suite de tags NFC et construit en mémoire une liste de tokens (programme). Les LEDs sont en bleu.
  • Mode Exécution — le robot interprète les tokens et pilote ses actionneurs (moteurs, LEDs). Les LEDs passent au vert.

La transition entre les deux modes est déclenchée par une détection de "flip" : retourner physiquement le robot est détecté par l'IMU (BMI270 + filtre de Madgwick). C'est l'interaction clé qui remplace tout écran ou bouton.

💡

En l'absence de programme scanné, le robot entre en mode idle avec un effet arc-en-ciel sur les LEDs. Après IDLE_EXECUTE_MS (10 s) sans activité NFC, l'exécution est déclenchée automatiquement.

Architecture

Le firmware est structuré autour de plusieurs modules indépendants qui communiquent via des structures de données partagées et FreeRTOS.

Flux de données

Tâches FreeRTOS

L'ESP32 est dual-core. Le firmware utilise FreeRTOS pour exécuter la lecture IMU en parallèle de la logique principale :

TâcheCorePrioritéRôle
loop() (Arduino) Core 1 Normale FSM principale — scan NFC, exécution programme
Task_imu (imu_handler) Core 0 Haute Lecture BMI270 en continu, calcul Madgwick, mise à jour de _data.flip
Task_prog (runProgram) Core 1 Normale Exécution de l'interpréteur (démarrée dynamiquement au flip-back)
⚠️

Les variables partagées entre le core IMU et la boucle principale (_data.flip, tagContents, tagsReadCount) sont accédées sans verrou dans la version actuelle. C'est acceptable ici car les accès sont atomiques sur les types natifs d'ESP32 et la fréquence d'écriture IMU est suffisamment lente.

Firmware

Le firmware est géré par PlatformIO. La configuration cible l'environnement esp32dev avec le framework Arduino. La partition flash utilise un schéma personnalisé (min_spiffs.csv) pour maximiser l'espace programme.

Dépendances (platformio.ini)

ini — platformio.ini
[env:esp32dev]
platform = espressif32
board    = esp32dev
framework = arduino
board_build.partitions = boards/min_spiffs.csv

lib_deps =
    adafruit/Adafruit PN532@^1.3.4          ; NFC PN532 SPI
    adafruit/Adafruit_VL53L0X@^1.2.4       ; ToF distance
    madhephaestus/ESP32Servo@^3.0.9        ; Servo (héritage)
    thewknd/VEML6040@^0.3.2               ; Capteur couleur RGBW
    adafruit/Adafruit NeoPixel@^1.15.1    ; LEDs WS2812
    sparkfun/SparkFun BMI270@^1.0.3        ; IMU accéléro+gyro

La librairie SCServo est embarquée localement dans lib/SCServo0009/ (protocol SCS half-duplex UART) car elle n'est pas disponible sur le registre PlatformIO.

Utilisation (Scan → Exécution)

  1. Flasher le firmware via pio run -e esp32dev -t upload et ouvrir le moniteur série à 115 200 bauds.
  2. Retourner le robot (flip) pour entrer en mode scan — les LEDs passent en bleu fixe.
  3. Approcher les briques NFC dans l'ordre voulu. Le premier tag doit obligatoirement être START. Chaque scan valide flash les LEDs en vert 100 ms, un tag invalide flash en rouge.
  4. À la fin de la séquence, retourner le robot dans sa position normale (flip back) — les LEDs passent en vert et l'interpréteur démarre.
  5. À la fin de l'exécution, le programme est effacé de la RAM et le robot repasse en mode idle (arc-en-ciel).

Langage NFC

Le "langage" Wood2Bot est une liste ordonnée de tokens, chacun écrit sur une brique NFC physique sous forme de texte NDEF. La grammaire est volontairement simple et inspirée des structures impératives classiques.

Format attendu des tags

Le firmware lit un enregistrement NDEF de type "Text" et compare le contenu (insensible à la casse, normalisé en majuscules en interne). Convention : 1 tag = 1 token, en texte brut, en MAJUSCULES.

ℹ️

L'UID de chaque tag est mémorisé pour éviter le re-scan accidentel du même tag physique dans une session. Seul le dernier tag scanné peut être re-détecté sans conséquence (tenu en place).

Les conditions utilisent le format : IF DISTANCE <cmp> <val_mm> ou IF COLOR = <couleur>. Les valeurs de distance doivent être des entiers d'au moins 2 chiffres (ex : 50, 150, 300).

Interpréteur

L'interpréteur est implémenté dans src/parser/parser.cpp. Il parcourt le vecteur tagContents de gauche à droite via un pointeur d'instruction entier i, avec gestion de la récursivité via une pile (LoopFrame).

Machine d'états du scanner

La validation de la séquence NFC se fait dans isValidTag(previousType, currentType). Chaque token est classifié en type logique avant validation :

Token(s)Type interne
MOVE_F, MOVE_B, TURN_L, TURN_RACTION
IF, REPEAT, ELSE, END_IF, END_REPEATtel quel
DISTANCE, COLORtel quel
=, <, >tel quel (comparateur)
RED, GREEN, BLUE, NOMBREVALUE
✅

Après un START ou une ACTION, seules les transitions vers ACTION, IF ou REPEAT sont autorisées. La validation garantit une grammaire minimale correcte avant même l'exécution.

Fonctionnement du parser

Le parser de runProgram() gère :

  • Actions : appels directs motors→moveForward(), motors→rotateByDegrees(), etc.
  • IF/ELSE/END_IF : évaluation de condition via evalCond(), saut au ELSE ou END_IF via findMatchingElseOrEndIf() (respecte l'imbrication).
  • REPEAT N / REPEAT condition : pile de LoopFrame qui mémorise l'index de retour et le compteur. Les boucles conditionnelles réévaluent le capteur à chaque itération.
c++ — logique IF simplifiée (parser.cpp)
if (T == "IF") {
  Cond c;
  bool ok = parseDistanceCond(i, c) || parseColorCond(i, c);
  if (!ok) { /* skip to END_IF */ }
  else if (!evalCond(c)) {
    i = findMatchingElseOrEndIf(i); // saute le bloc IF
    if (i < tagsReadCount && tagContents[i] == "ELSE") i++;
  }
}

Les tournants sont gérés par motors→rotateByDegrees(±90.0f, imu) qui utilise un contrôleur PID sur l'angle de lacet (yaw) calculé par le filtre de Madgwick.

Flasher les tags NFC

Pour écrire vos propres briques NFC, utilisez l'application NFC Tools (disponible sur Android et iOS). L'objectif est d'écrire un NDEF Text Record contenant exactement le token attendu.

Procédure conseillée

  1. Ouvrir NFC Tools et aller dans l'onglet Write.
  2. Appuyer sur Add a record → choisir Text.
  3. Dans le champ Content, saisir le token exact (ex : MOVE_F, START, 150).
  4. Appuyer sur Write puis approcher le tag NFC du téléphone.
  5. Vérifier via l'onglet Read que le contenu est bien le token attendu.
⚠️

Utiliser un seul record Text par tag NFC. Éviter les espaces en début/fin. Garder les tokens en MAJUSCULES. Pour les distances, écrire uniquement les chiffres (ex : 150, pas 150mm).
